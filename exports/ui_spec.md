# HH Monitor UI v1 (Draft for Alignment)

## Контекст
- Продукт: локальный GUI monitor/parser для hh.ru вакансий.
- Главная задача пользователя: быстро видеть изменения в выдаче и управлять параметрами сбора без CLI.
- Платформа: desktop-first web-like UI (с адаптацией до mobile).

## User Flow (критический путь)
1. Открыть приложение и проверить статус текущего сканирования.
2. Скорректировать фильтры (query/mode/pages/keywords).
3. Запустить deep-run.
4. Сразу увидеть KPI по new/updated/removed/errors.
5. Провалиться в конкретную вакансию и открыть исходник на hh.ru.

## Deep-run UX логика (PO-approved)
- В deep-run попадают не все вакансии, а только кандидаты из fast-pass с приоритетом.
- Базовые правила отбора:
  1. `new` и `updated` со значимыми изменениями (salary/title/company/snippet).
  2. Совпадение с high-signal include keywords (например, CEO/COO/Chief of Staff).
  3. Вакансия не deep-сканировалась недавно (cooldown/freshness окно).
  4. Ручной pin пользователем (`force deep-run`) всегда поднимает в очередь.
  5. Очередь режется лимитом запуска (например, `12/run`) с сортировкой high -> medium -> low.
- Где отображается:
  - Основная таблица: весь fast-pass поток (все найденные изменения).
  - Блок `Очередь deep-run`: список, что будет открываться глубоко, с причиной отбора и текущим статусом.
  - Блок `Результаты deep-run`: отдельный список только deep-полей (`description`, `activity_text`, full salary, diff-изменения).
- UX-действия пользователя:
  - В основной таблице: `Pin to deep`, `Skip`, `Run now` (для приоритизации конкретной вакансии).
  - В очереди deep-run: `Run now` (поднять вверх очереди), `Skip` (исключить из текущего прогона).
  - В результатах deep-run: только чтение (аналитика изменений), без изменения очереди.

## Независимые мнения ролей

### UI/UX Engineer
- Нужна структура `Control Panel -> KPI -> Data Table -> Details`, чтобы приоритет действий был очевиден.
- KPI-карточки должны быть компактными, без визуального шума, с явной семантикой статусов.
- На mobile таблица должна перестраиваться в карточки-строки без потери читаемости.
- Обязательные состояния на следующей итерации: empty/loading/error/success.
- По решению пользователя тёмная тема включается в v1: нужен ручной toggle + поддержка `prefers-color-scheme`.
- Явные action-кнопки в строке сокращают когнитивную нагрузку: не нужно открывать карточку для простых решений.

### Black Hat Critic
- Риск перегруза: слишком много параметров в левой панели может отпугнуть при первом запуске.
- Риск ложного доверия: пользователь может не увидеть, что данные устарели, если нет явного `last updated` timestamp.
- Риск ошибки действия: кнопка `Запустить deep-run` может запускаться повторно без lock-состояния.
- Риск тёмной темы: при ручном переключении возможны неконсистентные контрасты тегов и вторичных кнопок.
- Риск конфликтов действий: пользователь может нажать `Skip` во время `in progress`.
- Решение: доработать (добавить блок freshness + disabled/loading состояния кнопок + контраст-аудит для dark/light + lock для `in progress`).

### QA
- Для релизной готовности не хватает формализованных acceptance criteria по состояниям и адаптиву.
- Минимальный smoke: старт/пауза run, сохранение фильтров, отрисовка таблицы, открытие URL.
- Регрессионный фокус: фильтры include/exclude, сортировка таблицы, корректность статусов new/updated/removed.
- Дополнительно для тёмной темы: проверить toggle, персистентность выбора темы, контраст текста/тегов/CTA в dark.
- Для новых UX-действий: проверить `Pin to deep/Run now/Skip`, включая гонки состояния и блокировку конфликтных кликов.
- Текущий QA verdict для макета: `go with known issues` (только для дизайн-драфта).

### Security Officer
- Возможны ПДн в логах/экспортах (контакты/имена в описаниях вакансий), нужны маскирование и контроль доступа к файлам.
- Для UI обязателен безопасный путь к отчётам и явные предупреждения перед экспортом/открытием внешних ссылок.
- Для РФ-контекста: предварительно релевантны 149-ФЗ + 152-ФЗ (если сохраняются ПДн).
- Риск блокирующий для прод-версии: отсутствие настройки retention/очистки логов.
- Для темы: хранить только preference (`light|dark`) в `localStorage`, без пользовательских данных.

### PO (решающая роль)
- Выбран вариант экрана: `операционный дашборд` с левым контроллером параметров и центральной таблицей изменений.
- Причина выбора: максимальная скорость принятия решений для пользователя при минимуме кликов.
- Что делаем в следующем спринте UI:
  1. Добавить `last run / freshness` и lock/loading у run-кнопок.
  2. Добавить пустые/ошибочные состояния таблицы и KPI.
  3. Вынести `quick filters` в верхнюю строку для fast-режима.
  4. Включить тёмную тему в v1 с auto-detect + ручным переключателем.
  5. Добавить явные строковые действия `Pin to deep`, `Run now`, `Skip` в fast-таблицу и deep-очередь.

## UI Tokens (текущий драфт)
- Цвета: `#11221d` (text), `#0f766e` (accent), `#f59e0b` (secondary accent), `#f4f7f6` (bg).
- Типографика: `Sora` (headings), `IBM Plex Sans` (body).
- Радиусы: 10/12/14/18.
- Motion: enter 260-340ms ease-out; interactive 120ms ease-in-out.
- Themes:
  - `light`: базовая палитра мониторинга.
  - `dark`: глубокий графитово-зелёный фон, высококонтрастные статусы и CTA.

## Gate-check (предрелизный дизайн-контроль)
- UI Release Gate: `PASS` для драфт-макета.
- Contrast Audit: `PASS` (первичный ручной аудит).
- Visual Snapshot: `PASS` (локальный осмотр).
- UAT Checklist: `PASS` для текущего сценария (без edge-состояний).
- Blocking Issues: нет для согласования концепта; есть для production-ready реализации.
